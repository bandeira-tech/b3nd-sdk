the data protocol

B3ND.

A URI-based protocol for data that belongs to you. Four operations. Client-side encryption. Schema-agnostic.

Protocol

A small protocol with a large guarantee.

Everything B3nd does, on one screen. Four operations over URI-addressed data. Schema-agnostic by design — B3nd doesn't care what your data looks like. It cares who it belongs to.

mutable://accounts/{pubkey}/profile

immutable://

Write-once records. Logs, posts, receipts.

hash://

link://

Mutable references to other URIs.

msg://

Message envelopes for batch writes.

Code

See it. Ship it.

The API surface is minimal by design. If you can write a fetch call, you can write to B3nd.

write 
// Store data at a URI you control
await client.receive([
  "mutable://accounts/alice/profile",
  { name: "Alice", bio: "Builder" }
]);
read 
// Read it back — any app, same address
const result = await client.read(
  "mutable://accounts/alice/profile"
);

console.log(result.record.data);
// { name: "Alice", bio: "Builder" }
any app, same data 
// A different app reads the same URI
const profile = await otherApp.read(
  "mutable://accounts/alice/profile"
);

// Same address. Same data. Your data travels with you.

Ownership

Ownership is not a feature. It's the protocol.

Most platforms promise you own your data, then store it behind their access controls. B3nd makes anything else structurally impossible.

Traditional platforms sit between users and their data. B3nd puts data at URI addresses you control — any application can read the same address, and switching apps doesn't mean starting over.

Privacy

Privacy through encryption, not permission.

Data is encrypted on the client before it leaves your device. The server stores ciphertext. It cannot read what it cannot decrypt.

Public
